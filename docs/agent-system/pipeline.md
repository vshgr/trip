# Агентский пайплайн для фич и изменений

Этот документ описывает рабочий процесс для каждой новой продуктовой задачи в Trip.

## 1. Git gate

Перед началом новой задачи нужно зафиксировать состояние репозитория:

```bash
git branch --show-current
git status --short --branch
```

Если текущая ветка содержит незакоммиченные или незапушенные изменения, нельзя молча переключаться на `main`. Нужно спросить пользователя, что делать:

- закоммитить и запушить текущую работу;
- продолжить работу на текущей ветке;
- отложить текущую работу и начать новую ветку.

Если рабочее дерево чистое и пользователь хочет новую задачу, стартуем от свежего `main`:

```bash
git switch main
git pull --ff-only origin main
git switch -c codex/<task-slug>
```

Для автоматизации:

```bash
scripts/codex-start-feature.sh "название задачи"
```

## 2. Роутинг ролей

Оркестратор выбирает роли по характеру задачи.

| Роль | Когда нужна | Основной результат |
| --- | --- | --- |
| Бизнес-аналитик | Всегда для фич и заметных изменений | Требования, сценарии, acceptance criteria |
| UX-дизайнер | Есть экран, навигация, форма, состояние или текст | UX-решение, место в интерфейсе, состояния |
| Frontend | SwiftUI, клиентская логика, визуальные состояния | Реализация UI и клиентского поведения |
| Backend | Модели, сохранение, миграции, API, виджетные данные | Контракты данных и серверная/хранилищная логика |
| QA | Всегда | Corner cases, регрессии, проверочный план |

## 3. Последовательность

1. Оркестратор: понять задачу, проверить git, создать ветку.
2. Бизнес-аналитик: описать цель, границы, сценарии и критерии приемки.
3. UX-дизайнер: выбрать место в приложении и описать интерфейсные состояния.
4. Frontend/Backend: реализовать изменения в нужных слоях.
5. QA: проверить сценарии, баги, пустые состояния, ошибки данных, регрессии.
6. Оркестратор: запустить релевантные проверки и отчитаться.

## 4. Definition of Done

Задача считается готовой, когда:

- изменения находятся в отдельной ветке `codex/*`, если пользователь не выбрал текущую ветку;
- требования и UX-решение учтены в реализации;
- данные не ломают существующие поездки, расходы, маршрут Europe Trip и виджет;
- проверены пустые, граничные и ошибочные состояния;
- финальный ответ содержит краткое описание изменений и результат проверок.
