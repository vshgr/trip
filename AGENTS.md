# Agent Workflow

Эти инструкции применяются к работе Codex в этом репозитории.

## Когда включать пайплайн

Запускай агентский пайплайн, когда пользователь предлагает новую фичу, изменение поведения, доработку интерфейса, исправление комплексного бага или просит "сделать", "добавить", "поменять", "улучшить".

Для мелких служебных просьб вроде проверки `git status`, просмотра файла или ответа на вопрос пайплайн не нужен.

## Git-подготовка перед новой задачей

Перед началом реализации новой фичи или изменения:

1. Определи текущую ветку:

   ```bash
   git branch --show-current
   ```

2. Проверь незакоммиченные и незапушенные изменения:

   ```bash
   git status --short --branch
   ```

   Если есть незакоммиченные изменения, staged changes, untracked files, `ahead` commits или локальная ветка без upstream с локальными коммитами, остановись и спроси пользователя:

   - закоммитить и запушить текущую работу;
   - продолжить работу на текущей ветке;
   - отложить текущую работу и начать новую ветку.

3. Если пользователь выбирает новую ветку, переключись на `main`:

   ```bash
   git switch main
   ```

4. Подтяни актуальный `main`:

   ```bash
   git pull --ff-only origin main
   ```

   Если fast-forward невозможен, остановись и объясни конфликт или расхождение.

5. Создай новую ветку от свежего `main` с префиксом `codex/`:

   ```bash
   git switch -c codex/<short-task-slug>
   ```

Можно использовать скрипт:

```bash
scripts/codex-start-feature.sh "краткое название задачи"
```

## Агентский состав

Перед реализацией определи, какие роли нужны для задачи. Минимально почти всегда нужны бизнес-аналитик и QA. Остальные роли подключай по области изменений.

Роли описаны в `docs/agent-system/roles/`:

- `business-analyst.md` - требования, сценарии, ограничения, критерии приемки.
- `ux-designer.md` - место в интерфейсе, UX-поведение, состояния, тексты, доступность.
- `frontend.md` - SwiftUI/iOS реализация и клиентская архитектура.
- `backend.md` - данные, сохранение, миграции, интеграции, виджетные контракты.
- `qa.md` - баги, corner cases, регрессии, тестовый план.

## Порядок работы над фичей

1. Оркестратор кратко пересказывает задачу и выбирает нужные роли.
2. Бизнес-аналитик фиксирует цель, пользовательские сценарии, ограничения и acceptance criteria.
3. UX-дизайнер определяет, куда встроить изменение, как оно выглядит и какие состояния нужны.
4. Frontend-агент реализует клиентскую часть, если затронут SwiftUI или UI-состояния.
5. Backend-агент реализует данные, хранение, интеграции, миграции и контракты, если они затронуты.
6. QA-агент проверяет баги, corner cases, регрессии и недостающие состояния.
7. Оркестратор проводит финальную проверку: `git diff`, релевантная сборка/тесты, краткий отчет пользователю.

## Формат ответа перед большой реализацией

Если задача больше маленького исправления, перед изменениями дай пользователю короткий план:

- выбранные роли;
- что будет сделано;
- какие файлы или подсистемы вероятно будут затронуты;
- чем будет проверено.

Не спрашивай подтверждение на очевидные шаги, если нет риска потерять работу или изменить продуктовый смысл. Если есть незакоммиченная/незапушенная работа, спрашивать обязательно.
